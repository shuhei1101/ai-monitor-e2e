# 設計変更影響調査 チェックシート

設計変更（機能追加・方式変更・廃止）の反映が全影響範囲に行き渡り、デッド設計が残っていないかを確認するシート。
項目は実行順で、前半（判定一覧の作成まで）は修正の着手前に、後半は修正の完了時に確認する。

## チェック項目

| 項目 | 補足 |
| --- | --- |
| 変更する概念を 1 文で言語化した | 例: 「イベント → セッションの解決方式を台帳照合に変える」 |
| その概念に触れる全要素を grep で列挙した | 論理名と物理名の両方で検索する（処理ステップの呼び出し先リンクにより、論理名 grep で呼び出し元も全ヒットする） |
| 列挙した各要素に 維持 / 修正 / 廃止 を判定した一覧を作った | 複数ターンに分けて反映する場合は、この一覧を先に提示してから着手する |
| 「廃止」判定の要素を削除した | デッド設計を残さない（履歴は git log に任せる） |
| 同じ問いに答える機構が 2 つ残っていない | 旧方式と新方式の併存・責務の重複がないことを確認する |
| 上流 → 下流の全レイヤーに反映した | シナリオ → 結合 → モジュール構成 →（テンプレート / 規約 / CLAUDE.md） |
| dev-kit 規約との矛盾がない | 言語 / テスト / ツール系の一般論に触れた場合は該当ルールも同じターンで修正する |
