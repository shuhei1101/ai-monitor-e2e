# ai-monitor テンプレート: 設定

プロジェクトの **設定値 / 環境変数** を集約する書式。
`settings.yaml` / `settings.py` / `.env` などで管理される値の一覧を、実装ファイルを開かなくても把握できる形にまとめる。

## 前提

- 設定値は設定ファイル（`settings.yaml` / `settings.py` 等）に集約する。
  設定ファイルは git 管理外にし、サンプル（`*.example`）をコミットする
- 設定ファイルが git 管理外のため、**秘匿情報**（API キー / DB パスワード等）も設定ファイルに書いてよい。
  yaml が使えるプロジェクトは極力 `settings.yaml` に収める
- 環境ごとの差分は `settings.{環境}.yaml`（例: `settings.test.yaml` / `settings.prod.yaml`）に書き、共通の `settings.yaml` の同名キーを上書きする
- アプリは起動時に `settings.yaml`（共通）→ `settings.{環境}.yaml`（環境差分）→ 同名の環境変数 の順に読み込み、後のものが同名キーを上書きする（読み込み制御はアプリ側で実装する）
- 環境名は環境変数（例: `{アプリ名}_ENV`）または起動引数で指定する。
  未指定なら共通の `settings.yaml` のみ読み込む

## 配置

- 常に **フォルダ + サブシステム別ファイル** で配置する（1 サブシステムしかなくても同じ）
  - 起点フォルダ: `設計図/設定/`
  - 各ファイル: `設計図/設定/{サブシステム名}.md`（例: `設計図/設定/バックエンド.md` / `設計図/設定/フロントエンド.md` / `設計図/設定/AI サーバー.md`）
- 1 ファイル運用は禁止（将来サブシステム追加時にファイル名変更が必要になり拡張性が悪いため）

## セクション一覧

| セクション | サブセクション | 必須or条件 | 補足 |
| --- | --- | --- | --- |
| `## ファイル一覧` | - | 必須 | どのファイルで何を管理するかの索引 |
| `## {ファイル名}` | - | ファイル一覧の各行につき 1 | キー一覧表 |

## `冒頭リード`

### 記述例

```markdown
# 設定: {サブシステム名}

`{サブシステム名}` の設定値一覧。

- 実装は `settings.py` / `settings.yaml` を参照。サンプルは `settings.yaml.example`
```

### 補足

- 実装ファイル・サンプルファイル（`settings.yaml.example` 等）へのポインタを冒頭に置く

## `## ファイル一覧`

### 記述例

```markdown
## ファイル一覧

| ファイル | 管理する値 | 補足 |
| --- | --- | --- |
| `settings.yaml` | 共通の設定値（秘匿情報含む） | git 管理外。サンプルは `settings.yaml.example` |
| `settings.test.yaml` | テスト環境の差分 | `settings.yaml` の同名キーを上書き |
| `settings.prod.yaml` | 本番環境の差分 | `settings.yaml` の同名キーを上書き |
```

### 補足

**カラム:**
`ファイル / 管理する値 / 補足`（3 列）

**ファイル列:**
- リポジトリルートからの相対パス・ホーム起点の絶対パス（`~/.config/{アプリ名}/settings.yaml` 等）・ファイル名のいずれか
- 環境差分ファイルは拡張子の前に環境名を入れる（`settings.{環境}.yaml`）。
  全環境共通なら `settings.yaml` のみ
- 設定ファイルを持たないサブシステムは表を出さず「なし」と記載

**管理する値列:**
- そのファイルが担う値の種類を 1 行で（`設定値（秘匿情報含む）` 等）

**補足:**
- 新規ファイル追加時は本表に 1 行追加 + 対応する `## {ファイル名}` セクションを新設

## `## {ファイル名}`

各設定ファイルのキー一覧。
ファイル一覧の各行につき 1 つの H2 を作成する。

### 記述例（settings.yaml）

````markdown
## settings.yaml

| 論理名 | キー | 型 | 必須 | 既定値 | 説明 | 補足 |
| --- | --- | --- | --- | --- | --- | --- |
| GitHub Token | `github_token` | `str` | ✅ | - | GitHub API の認証 PAT | 秘匿。欠落時は起動エラー |
| 待受ポート | `port` | `int` | - | `8765` | HTTP API の待受ポート | - |
| ログレベル | `logging.level` | `"DEBUG"` \| `"INFO"` \| `"WARN"` | - | `INFO` | ログ出力の下限レベル | - |
| 監視対象プロジェクト | `projects[]` | `list` | ✅ | - | 監視するプロジェクトの一覧 | 子キーは下の行で展開 |
| 監視対象プロジェクト | `projects[].name` | `str` | ✅ | - | プロジェクト名 | - |
| 監視対象プロジェクト | `projects[].repo` | `str` | ✅ | - | GitHub リポジトリ（`owner/name`） | - |
| 監視対象プロジェクト | `projects[].local_path` | `str` | ✅ | - | クローンのルート絶対パス | - |
````

### 記述例（settings.prod.yaml・環境差分）

````markdown
## settings.prod.yaml

| 論理名 | キー | 値 | 説明 | 補足 |
| --- | --- | --- | --- | --- |
| ログレベル | `logging.level` | `WARN` | 本番はノイズを減らす | - |
| LLM タイムアウト | `openai.timeout_ms` | `30000` | 本番は余裕を持たせる | - |
````

### 記述例（.env・yaml を持たないプロジェクトの代替）

````markdown
## .env

| 論理名 | キー | 型 | 必須 | 既定値 | 説明 | 補足 |
| --- | --- | --- | --- | --- | --- | --- |
| GitHub Token | `GITHUB_TOKEN` | `str` | ✅ | - | GitHub API の認証 PAT | 秘匿。欠落時は起動エラー |
````

### 補足

**カラム:**
`論理名 / キー / 型 / 必須 / 既定値 / 説明 / 補足`（7 列）
- 環境差分ファイル（`settings.{環境}.yaml`）は `論理名 / キー / 値 / 説明 / 補足`（5 列）。
  共通の同名キーを上書きする値だけを書く

**論理名列:**
- 業務用語（例: `DB ホスト` / `LLM モデル`）

**キー列:**
- 階層構造はドットで結ぶ（`logging.level`）。
  フラットな設定は 1 語（`port`）
- オブジェクトの子: `parent.child` / 配列要素の子: `parent[].child`。
  親行の補足列に「子キーは下の行で展開」と書く

**型列:**
- プロジェクト言語の型表記（`str` / `int` / `bool` / `list[str]` 等）
- 取りうる値が限定されるキーはリテラルの列挙で書く（`"DEBUG"` \| `"INFO"` \| `"WARN"`）

**必須列:**
- `✅` / `-`。
  `✅` で欠落時の挙動を補足列で明示（`起動時にエラー` / `該当機能を無効化` 等）

**既定値列:**
- 省略時に適用される値。
  必須キーは `-`

**補足:**
- 秘匿キーは補足列に `秘匿` と明記し、実値は書かない
- 廃止予定 / 移行中のキーは補足列で明示（`廃止予定: v2.0 で削除`）

## 表のルール（全テンプレ共通）

`docs/wiki/テンプレート/テンプレート.md` の「表のルール」に従う。
- 補足列: 最右に必須
- 連続同値: 省略せず実値を書く
- 該当なし: `-`
