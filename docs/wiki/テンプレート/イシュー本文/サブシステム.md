# ai-monitor テンプレート: サブシステム

`layer:subsystem` が付いた Issue の本文書式。
1 対象システム（BE / FE / AI サーバー 等、`scope:*` で識別）ごとに 1 Issue を切る。

subsystem では **その対象システム内での実装 SA（機能要件 + 非機能要件）** を確定する。
UI 設計 / システム方式設計（SS） / 実装計画 / ドキュメント更新 は **Draft PR 側** で扱う。

## 担当セクション一覧

| No | セクション | サブセクション | 必須or条件 | 担当 |
| --- | --- | --- | --- | --- |
| 1 | `## 前提条件` | - | 依存 Issue/PR がある場合 | subsystem-issue-triager |
| 2 | `## 概要` | - | 必須 | subsystem-issue-triager |
| 3 | `## 背景` | - | 〃 | 〃 |
| 4 | `## 現状` | `### 関連実装コード` | 〃 | subsystem-issue-triager |
| 5 | 〃 | `### 関連テスト` | 〃 | 〃 |
| 6 | 〃 | `### 関連 Issue/PR` | 〃 | 〃 |
| 7 | 〃 | `### 関連ドキュメント` | 〃 | 〃 |
| 8 | 〃 | `### 既存テスト実行結果` | テストが実行可能な場合 | 〃 |
| 9 | 〃 | `### 再現手順` | バグ報告の場合のみ | 〃 |
| 10 | `## システム要件（SA）` | `### 機能要件` | 必須 | subsystem-issue-triager |
| 11 | 〃 | `### 非機能要件` | 性能/セキュリティ/運用要件がある場合 | 〃 |
| 12 | 〃 | `### スコープ外` | 誤解防止のため可能なら書く | 〃 |

> **UI 設計 / システム方式設計 / PoC 結果 / 実装計画 / ドキュメント更新** は Issue 本文では扱わない（Draft PR + `docs/wiki/` で扱う、docs 先行 → 実装の順）。

## `## 前提条件`

### 記述例

```markdown
## 前提条件

| No | 種類 | 対象 | 状態 | 補足 |
| --- | --- | --- | --- | --- |
| 1 | Issue | #93（兄弟 subsystem: バックエンド） | 完了 | BE の API 契約確定後に FE 着手 |
| 2 | PR | #220（DB スキーマ変更） | merged | - |
```

### 補足

- **親 story は書かない**（GitHub の Sub-issue リンクで表現済み、`get_issue.parent` で取得可）
- **親子リンクで表現できない追加依存のみ**記載する（兄弟 subsystem との順序 例: BE → FE / AI サーバー → BE、別 PR との依存 等）
- 依存する Issue / PR がない場合も本セクションは残し、本文に「なし」と記載する

## `## 概要`

### 記述例

```markdown
## 概要

プロフィール編集機能のフロントエンド側実装（画面遷移・入力フォーム・バリデーション・API 呼び出し）を担当する。
```

### 補足

- 「この対象システムでの担当範囲」を 1〜3 行にまとめる
- 親 story のうち自 subsystem がどこを担当するかを明示
- `scope:*` ラベルと一致する対象システム名を明示（例: フロントエンド / バックエンド / AI サーバー）

## `## 背景`

### 記述例

```markdown
## 背景

親 story「プロフィールを編集して保存する」のフロントエンド担当。
既存の閲覧画面（`/profile/view`）に編集動線を追加し、`/profile/edit` 画面を新設する。
```

### 補足

- 親 story の抜粋 + 自 subsystem 固有の状況を書く
- 既存実装との関係（既存画面を拡張 / 新規画面を追加 等）を明示

## `## 現状`

### `### 関連実装コード`

**概要:**

Issue が言及する領域のコードベースを調査し、現状の実装がどこにあるかを整理する。
既存実装の所在・役割を把握することで、後続フェーズが「何を変えるか」「どこに追加するか」の判断材料を持てるようにする。

**記述例:**

```markdown
### 関連実装コード

| No | ファイル | メソッド | 概要 | 補足 |
| --- | --- | --- | --- | --- |
| 1 | `pages/profile/view.html` | - | プロフィール表示画面 | 編集動線をここに追加 |
| 2 | `api/users/{id}.py` | `get_user` | GET エンドポイント | PUT は未実装 |
| 3 | `services/user_service.py` | `get_user` | ユーザー取得処理 | `update_user` なし |
```

**ファイル列:**
- リポジトリルートからの相対パスで書く

**メソッド列:**
- ファイル全体への参照ならメソッド列は `-`
- 特定メソッドなら関数名・メソッド名をそのまま記載

**概要列:**
- そのファイル / メソッドの役割を 1 行で要約

**補足:**
- 領域別コードベース調査の結果を表で整理
- 自 subsystem の担当範囲（`scope:*`）のコードに絞る

### `### 関連テスト`

**概要:**

`### 関連実装コード` で洗い出した実装に紐づくテストを調査して整理する。
既存テストが何をカバーしているか・していないかを可視化し、後続フェーズが「どのテストを変更すべきか」「どこに新規テストが必要か」を判断する素材にする。

**記述例:**

```markdown
### 関連テスト

| No | 分類 | ファイル | メソッド | 概要 | 補足 |
| --- | --- | --- | --- | --- | --- |
| 1 | E2E | `tests/e2e/profile_view.spec.ts` | - | 表示のみカバー | 編集テストなし |
| 2 | 単体 | `tests/services/test_user_service.py` | `test_get_user` | get_user の単体テスト | - |
```

**分類列:**
- `単体`: 単一の関数・クラス・メソッドを外部依存なしで検証するテスト
- `結合`: 複数のモジュールが連携するユースケースを通しで検証するテスト（外部依存は Mock 注入）
- `E2E`: ユーザー操作を起点に画面 → API → DB → 画面復帰までの一連動作を検証するテスト
- `外部疎通`: 外部サービス（LLM / 決済 / メール送信 等）への実通信が成立するかを検証するテスト（料金発生のためユーザー手動実行のみ）

**メソッド列:**
- ファイル全体実行ならメソッド列は `-`

**補足:**
- 「カバーしていない領域」は補足列で明示する（ギャップを後続フェーズに引き継ぐ）

### `### 関連 Issue/PR`

**概要:**

同じキーワード・同じ機能・同じファイルに関わる既存 Issue / PR を検索して整理する。
重複起票の発見、過去判断の参照、影響を受ける可能性のある直近 merged PR の警告を後続フェーズに引き継ぐ。

**記述例:**

```markdown
### 関連 Issue/PR

| No | 番号 | 状況 | 概要 | 補足 |
| --- | --- | --- | --- | --- |
| 1 | Issue#102 | open | 同じ要望、本 Issue に集約 | 集約後 close 予定 |
| 2 | Issue#145 | closed | 過去の同様要望（実装見送り） | 当時の判断理由を参考 |
| 3 | PR#220 | merged | `user_service.py` を直近で改修 | 影響注意 |
```

**番号列:**
- Issue は `Issue#xx`、PR は `PR#xx` で区別

**状況列:**
- `open` / `closed` / `merged` のいずれか

**補足:**
- `related-issue-finder` / `related-pr-finder` の戻り値を整形
- 過去にバグを埋め込んだ可能性のある merged PR は補足列に「影響注意」と明記

### `### 関連ドキュメント`

**概要:**

Issue の領域に関係する既存ドキュメント（Wiki / CLAUDE.md / Rules / agents）を洗い出して整理する。
後続の PR 側 UI/arch 設計フェーズが「どのドキュメントを更新すべきか」を判断する素材になる。

**記述例:**

```markdown
### 関連ドキュメント

| No | 分類 | ページ | 概要 | 補足 |
| --- | --- | --- | --- | --- |
| 1 | Wiki | `設計図/モジュール構成/ユーザー.md` | User 周辺のクラス構造 | 更新対象 |
| 2 | 〃 | `設計図/バックエンド結合/README.md` | 既存エンドポイント一覧 | PUT 追加で更新が必要 |
| 3 | 〃 | `設計図/ER図/ユーザー.md` | users テーブル定義 | bio カラム追記が必要 |
```

**分類列:**
- `Wiki` / `CLAUDE.md` / `Rules` / `agents` など

**補足:**
- 本 Issue で更新が必要そうなら補足列に明記
- PR 側 UI/arch フェーズが「docs 先行」で更新するときの素材になる

### `### 既存テスト実行結果`

**概要:**

`### 関連テスト` で洗い出したテストを実際に走らせた結果を記録する。
バグ報告 Issue では再現確認の証拠に、新機能追加 Issue では「現状動作する範囲」を把握して回帰検知の基準にする。

**記述例:**

```markdown
### 既存テスト実行結果

| No | 分類 | ファイル | メソッド | 概要 | 結果 | 補足 |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | E2E | `tests/e2e/profile_view.spec.ts` | - | プロフィール表示 E2E | ✅ | 編集テストは未作成 |
| 2 | 単体 | `tests/services/test_user_service.py` | `test_get_user` | get_user 単体テスト | ✅ | - |
```

**分類列:**
- `単体` / `結合` / `E2E` / `外部疎通` のいずれか

**メソッド列:**
- ファイル全体実行ならメソッド列は `-`

**結果列:**
- `✅`（pass）/ `❌`（fail）。落ちた場合は補足列にエラー要約

**補足:**
- テストが実行可能な場合のみ記入
- 外部疎通テストは料金発生のため自動実行禁止。手動実行が必要な旨を補足列に書くに留める

### `### 再現手順`

**概要:**

バグ報告 Issue で「どう操作すれば問題が再現するか」「期待動作と実際の動作のギャップ」を記録する。
後続の修正フェーズで再現確認・回帰テスト作成の基準になる。

**記述例:**

```markdown
### 再現手順

1. ログイン後、`/profile/edit` に遷移
2. 自己紹介欄に 600 文字を入力
3. 保存ボタンをクリック

期待: バリデーションエラー表示
実際: 500 Internal Server Error
```

**補足:**
- **バグ報告の場合のみ記入**。新機能追加 Issue では省略
- 番号付き箇条書きで再現手順、末尾に「期待 / 実際」を明示
- 再現しなかった場合もその旨を記録する（環境差・キャッシュなどユーザー環境の問題かもしれない）

## `## システム要件（SA）`

### `### 機能要件`

**概要:**

自 subsystem が「何ができるか」を機能単位で列挙する。
subsystem = 1 対象システム なので **対象システム列は不要**（親 Issue の `scope:*` で自明）。

**記述例:**

```markdown
### 機能要件

| No | カテゴリ | 要件 | 補足 |
| --- | --- | --- | --- |
| 1 | 編集機能 | プロフィール編集フォームで氏名・自己紹介・画像を編集できる | - |
| 2 | バリデーション | 氏名は 1〜50 文字（必須） | 入力直後にチェック |
| 3 | 〃 | 自己紹介は 500 文字以内（任意） | 入力直後にチェック |
| 4 | 〃 | アバター画像は 5MB 以下・PNG/JPG のみ | 選択直後にチェック |
| 5 | エラー表示 | 各項目のエラーは入力欄下にインラインで表示 | - |
| 6 | 〃 | 保存失敗時はページ上部にトースト表示 | - |
| 7 | 状態表示 | 保存処理中はボタン無効化 + スピナー表示 | 二重送信防止 |
| 8 | レスポンシブ対応 | デスクトップ・タブレット・スマホで利用可能 | ブレークポイント 768px / 1024px |
```

**カテゴリ列:**

代表的なカテゴリ一覧（参考、網羅ではない）:

| カテゴリ | 要件の例 |
| --- | --- |
| 作成機能 | フォーム入口 / デフォルト値 / POST エンドポイント |
| 編集機能 | 編集可能フィールドの UI / 楽観的ロック / PUT-PATCH エンドポイント |
| 削除機能 | 削除確認ダイアログ / DELETE / 物理 vs 論理削除 |
| 閲覧機能 | 一覧の見え方 / フィルタ・ソート UI / GET エンドポイント / ページネーションパラメータ |
| 検索機能 | 検索ボックス / オートコンプリート / 検索クエリ処理 / インデックス |
| 一括操作 | 複数選択 UI / バルクエンドポイント / トランザクション |
| バリデーション | クライアント即時バリデーション / サーバー最終防衛 / DB 制約 |
| エラー表示 | インラインエラー / トースト / ダイアログ |
| エラー返却 | HTTP ステータス / エラーコード / エラーレスポンス構造 |
| 状態表示 | ローディング / 空状態 / 成功後遷移 |
| 通知 | アプリ内通知 / メール送信 / プッシュ配信 |
| 権限・認可 | 表示切替 / 認可チェック・RBAC |
| 履歴・監査 | 操作履歴 / 監査ログ書き込み |
| インポート / エクスポート | ファイル選択 UI / バッチ処理 / 非同期処理 |
| 国際化 (i18n) | 言語切替 UI / タイムゾーン変換 / メッセージリソース |
| アクセシビリティ | キーボード操作 / スクリーンリーダー / ARIA |
| レスポンシブ対応 | モバイル対応 / ブレイクポイント |
| 生成機能 | AI 応答生成 / 要約 / 分類（AI サーバー系） |
| 既存機能への影響 | 既存 API / 画面 / DB スキーマへの波及 |

**カテゴリ運用ルール:**
- **上記はあくまで例**。プロジェクト固有 / Issue 固有のカテゴリは自由に追加してよい
- **該当しないカテゴリは書かない**
- **1 カテゴリに複数の要件が来ることは普通にあり**（例: バリデーションが 3〜5 項目）→ 1 要件 = 1 行、カテゴリ列は `〃` で省略

**補足:**
- 1 要件 1 行。条件分岐や複雑なフローは補足列または別 No に分割
- **対象システム列は不要**（subsystem = 1 対象システム、`scope:*` で自明）
- 自 subsystem の担当範囲外の要件は書かない（兄弟 subsystem の Issue で扱う）

### `### 非機能要件`

**概要:**

機能要件以外の品質要件（性能・セキュリティ・可用性・運用 など）を列挙する。
subsystem = 1 対象システム なので対象システム列は不要。
数値基準を明示して、後続フェーズが計測可能なテスト・設計判断に使える素材にする。

**記述例:**

```markdown
### 非機能要件

| No | カテゴリ | 要件 | 補足 |
| --- | --- | --- | --- |
| 1 | 性能 | 初期ロード 2 秒以内（3G 相当ネットワーク） | LCP 基準 |
| 2 | アクセシビリティ | WCAG 2.1 AA 準拠 | 主要導線のみ |
| 3 | ブラウザ対応 | 最新 Chrome / Safari / Firefox の 2 世代前まで対応 | - |
```

**カテゴリ列:**

代表的なカテゴリ一覧（参考、網羅ではない）:

| カテゴリ | 要件の例 |
| --- | --- |
| 性能 | 応答時間 / QPS / 同時接続数 / 初期ロード時間 / LCP |
| 可用性 | SLA / 稼働率 / 冗長化 / オフライン対応 |
| セキュリティ | 認証・認可 / 個人情報 / 暗号化 / XSS / CSRF |
| 監査・コンプライアンス | ログ保存期間 / 法令対応（GDPR 等） |
| 運用 | 監視 / アラート / バックアップ |
| インフラ制約 | DB コネクション / メモリ / ストレージ |
| 拡張性 | スケーラビリティ / 負荷対応 |
| 保守性 | ログレベル / エラーメッセージ品質 / コードスタイル |
| コスト | ランニング予算 / スケール増分 / AI 呼び出し単価 |
| 外部 API 制約 | レートリミット / 料金 / タイムアウト |
| 外部ライブラリ | ライセンス / バージョン固定 |
| 国際化 | タイムゾーン / 通貨表示 / 多言語対応の範囲 |
| アクセシビリティ | WCAG 準拠レベル |
| ブラウザ対応 | 対応ブラウザ / 対応バージョン |

**カテゴリ運用ルール:**
- **上記はあくまで例**。プロジェクト固有 / Issue 固有のカテゴリは自由に追加してよい
- **該当しないカテゴリは表の中で行を作らない**。非機能要件が 1 件もない場合（フロント側で該当なしのケース等）でもセクション自体は残し、本文に「なし」と記載する
- **1 カテゴリに複数の要件が来ることは普通にあり**（例: 性能に「応答時間」と「同時接続数」の 2 行 等）→ 1 要件 = 1 行、カテゴリ列は `〃` で省略

**補足:**
- 数値で測れる基準を補足列に明記（「速い」「重い」など主観表現は避ける）
- **対象システム列は不要**（subsystem = 1 対象システム）

### `### スコープ外`

**概要:**

本 subsystem では実装しない項目を明示する。誤解で実装スコープが膨らむのを防ぎ、関係者の期待値を揃える。
将来やる予定があれば後続 Issue 起票候補としても活用する。

**記述例:**

```markdown
### スコープ外

| No | 項目 | 理由 | 補足 |
| --- | --- | --- | --- |
| 1 | パスワード変更 | 別フロー（メール認証込み）が必要 | 別 Issue 起票予定 |
| 2 | メールアドレス変更 | メール認証フローが必要 | 別 Issue 起票予定 |
| 3 | プロフィール公開設定 | 将来要件、本リリース対象外 | - |
```

**理由列:**
- なぜ今回スコープ外なのかを 1 行で説明

**補足:**
- 「今回はやらない」項目を明示することで実装スコープの膨張を防ぐ
- 将来やる予定なら補足列に別 Issue 起票候補を書く
