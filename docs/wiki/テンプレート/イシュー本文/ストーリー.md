# ai-monitor テンプレート: ストーリー

`layer:story` が付いた Issue の本文書式。
1 ユースケース（1 シナリオ）を扱う親 Issue で使う。

story は **「何のユースケースを扱うか」を要件レベルで明示** するのが役割。

1 story = 1 シナリオファイル = 1 E2E テストファイル に 1:1 対応する。

## 担当セクション一覧

| No | セクション | サブセクション | 必須or条件 | 担当 |
| --- | --- | --- | --- | --- |
| 1 | `## 前提条件` | - | 必須 | story-issue-triager |
| 2 | `## 概要` | - | 必須 | story-issue-triager |
| 3 | `## 背景` | - | 〃 | 〃 |
| 4 | `## ユースケース要件` | - | 〃 | story-issue-triager が草案、ユーザー承認後に確定 |

## `## 前提条件`

### 記述例

```markdown
## 前提条件

| No | 種類 | 対象 | 状態 | 補足 |
| --- | --- | --- | --- | --- |
| 1 | Issue | #92（兄弟 story: 認証機能） | 完了 | 認証ユースケース完了後に着手 |
| 2 | PR | #180（Better Auth 導入） | merged | - |
```

### 補足

- **親 epic は書かない**（GitHub の Sub-issue リンクで表現済み、`get_issue.parent` で取得可）
- **親子リンクで表現できない追加依存のみ**記載する（別 epic / 別 PR / 兄弟 story との順序依存 等）
- 依存する Issue / PR がない場合も本セクションは残し、本文に「なし」と記載する

## `## 概要`

### 記述例

```markdown
## 概要

ログイン中ユーザーが自身のプロフィール項目（氏名・自己紹介・アバター画像）を編集・保存する。
```

### 補足

- 1 ユースケースを 1〜2 行でまとめる
- 「{アクター} が {何をする}」で書く

## `## 背景`

### 記述例

```markdown
## 背景

親 epic #90「プロフィール自己管理機能」の `## ユースケース一覧` No.1「プロフィール編集」に対応。
既存の閲覧機能（`/profile/view`）に対して編集動線を追加する必要がある。
```

### 補足

- 概要と重複しない（概要は「何をするか」、背景は「なぜ必要か」）
- なぜこの story を切り出したか（epic をなぜこの粒度で分けたか）を書く
- **親 epic Issue 番号 + `## ユースケース一覧` の UC 番号 + UC 名** の 3 点セットを 1 行入れる（例: `親 epic #90 の UC No.1「プロフィール編集」に対応`）
- **1 story = 1 UC**。複数 UC を持つ story は禁止（分割する）

## `## ユースケース要件`

**このユースケース固有の要件**を列挙する。エピックの `## 横断要件`（全 UC 共通）と役割分担する形。

### 記述例

```markdown
## ユースケース要件

| No | 要件 | 補足 |
| --- | --- | --- |
| 1 | 氏名を編集できる | 50 文字上限 |
| 2 | 自己紹介を編集できる | 500 文字上限 |
| 3 | アバター画像をアップロードできる | 5MB 上限、JPG/PNG |
| 4 | 保存時にバリデーションエラーをインライン表示 | フィールド直下に表示 |
| 5 | 同時編集を検知したら警告ダイアログを出す | epic 横断要件 #4 に基づく実装 |
| 6 | 保存成功時にトーストで通知 | 3 秒表示 |
```

### 補足

**要件列:**
- **ユーザー視点の 1 文** で書く（「{ユーザー} が {アクション} できる / される」）
- 実装用語（API / DB カラム / コンポーネント名）を避ける
- 実装側の詳細（バリデーション具体値・ライブラリ選定 等）は subsystem 側で扱うのでここでは書かない

**補足列:**
- 定数・上限値・振る舞いの追加情報
- 横断要件を参照する行は `epic 横断要件 #N に基づく` と明記（整合性の起点を残す）

**補足:**
- 1 行 = 1 要件。条件分岐は補足列 or 別 No に分割
- **横断要件（複数 UC 共通）は書かない**（それは epic 側の `## 横断要件`）
- 本セクションが Claude Code / 実装エージェントが実装計画を立てる元になる（1 story だけ読めば完結する状態を維持）
