# ai-monitor テンプレート: PR 本文 / ライブラリPoC

subsystem のライブラリ選定 PoC 用 Draft PR（`poc/{scope}/{ドメイン}/{UC名}/{lib名}`、base=master・マージせず close）の本文書式。
1 候補 = 1 PR。
PR タイトルは `PoC: {ライブラリ名}（#{subsystem issue 番号}）`（例: `PoC: langchain（#200）`）。

## セクション一覧

| セクション | サブセクション | 必須or条件 | 担当 |
| --- | --- | --- | --- |
| `## 紐づく Issue` | - | 必須（PR 作成時） | 発注元 worker（PR 作成者） |
| `## 発注元 PR` | - | 必須（PR 作成時） | 発注元 worker（PR 作成者） |
| `## 検証対象` | - | 必須（PR 作成時） | 発注元 worker（PR 作成者） |
| `## 調査結果` | `### 使い方の要点` / `### コード例` / `### 観点別評価` | 必須（PR 作成時） | 発注元 worker（PR 作成者） |
| `## 検証観点と結果` | - | 必須（観点・成功条件は PR 作成時に発注元が記入。実測値・判定・所感は検証実行後に library-poc-runner が記入） | 発注元 worker + library-poc-runner |
| `## 最小再現コード` | - | 必須（検証実行後に記入） | library-poc-runner |

## `## 紐づく Issue`

### 記述例

```markdown
## 紐づく Issue

- #200 プロフィール自動要約 BE サブシステム
```

### 補足

- 親 subsystem Issue 番号を書く（1 件のみ）

## `## 発注元 PR`

### 記述例

```markdown
## 発注元 PR

- #210
```

### 補足

- 発注元の subsystem PR 番号を書く（モニターのセッション解決と、候補比較の議論スレッドへ辿る入口）

## `## 検証対象`

### 記述例

```markdown
## 検証対象

| 項目 | 内容 | 補足 |
| --- | --- | --- |
| ライブラリ | langchain | - |
| 概要 | LLM アプリ構築フレームワーク（チェーン / ストリーミング） | 1〜2 行の要約 |
| バージョン | 0.3.x | 検証したバージョンを固定値で |
| ライセンス | MIT | 商用利用可否を確認した上で記入 |
| 公式 URL | https://github.com/langchain-ai/langchain | GitHub / npm / 公式サイトの主要 1 つ |
| 公式ドキュメント | https://python.langchain.com/docs/ | API リファレンスの URL |
| 既存 Wiki | `外部ライブラリ/langchain.md` | 同一ライブラリのページが既にあれば記載（なければ `-`） |
```

### 補足

- 候補 1 つ分だけ書く（比較表は発注元 PR の候補比較コメント側）
- **採用決定時に `外部ライブラリ/{lib名}.md`（書式は `テンプレート/外部ライブラリ.md`）へ転記する元情報**。
  - 概要 / バージョン / ライセンス / 公式 URL / 公式ドキュメントをここで揃えておく
- **既存 Wiki 行**: 同一ライブラリを既に一部採用済みで、別の使い方を検証するケースで記載する
  - poc-runner は既存ページの記載（採用済みバージョン・使用中のメソッド）も前提知識として読む
  - このケースの採用決定時は、新規ページ作成ではなく既存ページへのメソッド追記になる

## `## 調査結果`

library-researcher の調査成果のうち、検証に必要な情報の転記。

### 記述例

````markdown
## 調査結果

### 使い方の要点

- インストール: `pip install langchain langchain-openai`
- 初期化はプロバイダ別パッケージ（`ChatOpenAI`）を DI で注入する構成が可能
- ストリーミングは `.stream()` がトークン単位のイテレータを返す

### コード例

```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")
for chunk in llm.stream("プロフィールを要約して"):
    print(chunk.content, end="")
```

### 観点別評価

| 観点 | 評価 | 補足 |
| --- | --- | --- |
| ストリーミング対応 | ○ | `.stream()` あり |
| 型サポート | △ | 戻り値の型が広い |
| 依存の重さ | △ | 依存パッケージが多い |
````

### 補足

- 「この本文だけで PoC 実装に着手できる」ことが基準（発注元 PR のコメントを読み直さなくて済む粒度で書く）
- インストール手順（バージョン固定・ピア依存）と使う予定のメソッド / パラメータを押さえる（採用決定時に外部ライブラリ Wiki の `## インストール手順` / `## メソッド一覧` の元になる）

## `## 検証観点と結果`

観点・成功条件は発注元が PR 作成時に記入し、実測値・判定・所感は library-poc-runner が検証実行後に記入する。

### 記述例

```markdown
## 検証観点と結果

| 観点 | 成功条件 | 実測値 | 判定 | 補足 |
| --- | --- | --- | --- | --- |
| ストリーミング応答 | トークン単位で受信できる | トークン単位で受信 | ✅ | バックエンド結合のインターフェースに必要 |
| 応答遅延 | 1 リクエスト 3 秒以内 | 1.8 秒 | ✅ | - |
| DI 差し替え | Mock クライアントに差し替えて単体テストできる | Mock 注入で単体テスト成功 | ✅ | 型エイリアス DI 前提 |

**所感:**
- ストリーミング時の例外がラップされて原因が読みにくい。実装時はエラーハンドリングに注意
```

### 補足

- 成功条件は**数値 or 観測可能な基準**で書く（PR 作成時点では実測値・判定列は `-` のまま）
- 判定列: `✅` / `❌`（成功条件と同じ行で 1:1 対応）
- 所感には **採用判断・後続実装への申し送り**（ハマりどころ・副作用・制約）を箇条書きで残す

## `## 最小再現コード`

### 記述例

````markdown
## 最小再現コード

```python
# poc_stream.py — ストリーミング検証の核心部（全体は PR diff 参照）
llm = ChatOpenAI(model="gpt-4o-mini")
for chunk in llm.stream(PROMPT):
    on_token(chunk.content)
```

**diff の見どころ:** `poc_stream.py`（検証本体）と `poc_di.py`（DI 差し替え確認）の 2 ファイルだけ。
````

### 補足

- **10〜30 行の核心部だけ**抜粋（全文は PR diff で見られるので重複させない）
- 末尾に **diff の見どころ**（どのファイルが本体か）を 1〜2 行で案内
