# ai-monitor: コメント規約

## 宛先・送信者の書き方

- モニター宛: `@implementer` / `@architect` / `@reviewer` など（モニター名そのまま）
- ユーザー宛: GitHub ログイン名 `@{ユーザー名}` など
- 複数宛: カンマ区切り `@implementer, @{ユーザー名}`
- モニター名とユーザー名は被らないので `@` プレフィックスだけで識別可能

### ユーザー名（GitHub ログイン名）の取得方法

```bash
gh api user --jq '.login'
```

## コメントフォーマット

1 つのコメント内で議論を続ける。AI/ユーザーが交互に区切り線で追記していく。

```
> 🤖 @reviewer → @implementer

## L42 の null チェック追加
戻り値が `User` ではなく `User | null` です。null チェック追加してください。

---
> @{ユーザー名} → @reviewer

その指摘は不要、null は呼び出し側で弾いてます

---
> 🤖 @reviewer → @implementer

了解、取り下げます。
```

### 判別ルール

- ブロック先頭に `🤖` あり → AI 記載
- 先頭に `🤖` なし → 人間（ユーザー）記載
- 2 行目はタイトル（要点 1 行）
- `@{宛先}` を見て自分宛か判断（自身モニター名 or 自分のログイン名）

### 例パターン

| No | シーン | 追記の先頭（1 行目） |
| --- | --- | --- |
| 1 | AI → AI（モニター間） | `🤖 @reviewer → @implementer` |
| 2 | AI → ユーザー | `🤖 @reviewer → @{ユーザー名}` |
| 3 | AI → 複数宛 | `🤖 @architect → @implementer, @{ユーザー名}` |
| 4 | ユーザー → AI | `@{ユーザー名} → @reviewer` |
| 5 | ユーザー → 複数宛 | `@{ユーザー名} → @implementer, @architect` |

## ユーザーコメントへの対応

ユーザーのコメントは「指示」「質問」「提案」「議論」など意図が様々。
ユーザーの言葉を鵜呑みにせず、AI が**自分で推論**してから対応する。

### コメントの分類と対応パターン

| No | ユーザーコメントの種類 | AI の対応 |
| --- | --- | --- |
| 1 | 質問（「ここってどうなってる？」など） | 質問に答える追記のみ。本文は触らない |
| 2 | 明確かつ妥当な指示（AI も合意） | 即本文反映 → 「本文の `## XXX` を `Y` に修正しました」と短く追記 |
| 3 | 不明確な指示・複数解釈可能な指示 | ワンクッション: 「この理解で合っていますか？こう書き換えます」と確認の追記 → ユーザー合意後に反映 |
| 4 | 妥当性に疑問がある指示・提案 | 推論内容を返す: 「ご指摘の方法だと {懸念点}。代替案として {案} はいかがでしょうか？」 → 議論 → 結論が出たら反映 |
| 5 | 議論・相談（「これどう思う？」） | AI の意見を返す → 結論が出たら反映 |

### 共通ルール

- 本文反映後の応答は短く（「本文の `## XXX` を更新しました」程度）
- 本文反映時は変更前の本文記述は消すこと（常に「最新の確定版」に保つ。古い情報は残さない。）
- 部分書き換えは難しいので、各モニター担当セクションを丸ごと置換
- コメントは個別に Resolve しない（ユーザーが反映結果を確認できなくなるため）。
  - Resolve はフェーズ終了時にまとめて行う

## Resolve のタイミング

ユーザーが `フェーズ終了` ラベルを付与したタイミングで、AI が**自分宛のコメントを一括 Resolve** する。

### 精査対象（自分宛のコメント）

| No | コメント分類 | 自分宛か | 扱い |
| --- | --- | --- | --- |
| 1 | 自身のモニターが投稿したコメント | はい | 反映状況を確認 → 一括 Resolve 対象 |
| 2 | ユーザー → 自身のモニター宛のコメント | はい | 反映状況を確認 → 一括 Resolve 対象 |
| 3 | 他のモニターが投稿したコメント | いいえ | 無視（当該モニターのフェーズ終了時に Resolve される） |
| 4 | ユーザー → 他のモニター宛のコメント | いいえ | 無視 |

宛先は各コメント先頭の `@{宛先}` を見て判定する。

### フロー

1. ユーザーが `フェーズ終了` を付与
2. AI が起動 → 自分宛のコメント（上記 No.1 / 2）だけを精査
3. 自分宛の未反映コメントがあるか確認:
   - 全て本文反映済み → 自分宛コメントを一括 Resolve → 次のラベルへ進める
   - 未反映あり → ユーザーに「これは反映してよいですか？反映不要ですか？」と質問コメント → `assignee=ユーザー` で待機 → ユーザー回答後に再判定
4. Resolve は `gh api graphql -f query='mutation { minimizeComment(input: { subjectId: "...", classifier: RESOLVED }) { ... } }'` で実行

## 対応例

| No | ユーザーコメント | 分類 | AI の対応 | AI の追記内容 |
| --- | --- | --- | --- | --- |
| 1 | 「ライブラリは LangChain ではなく Anthropic SDK にして」 | 2 明確な指示 | 本文「採用ライブラリ」を更新（コメントは残す） | 「採用ライブラリを Anthropic SDK に変更しました」 |
| 2 | 「分割案 B でお願い」 | 2 明確な指示 | 本文「設計方針」を更新 + 子 Issue 作成（コメントは残す） | 「分割案 B を採用、子 Issue #45/#46 を作成しました」 |
| 3 | 「キャッシュ層、もう少しシンプルにできない？」 | 3 不明確な指示 | ワンクッションで確認 → ユーザー合意後に反映 | 「インメモリ LRU に変更しキャッシュサイズを 1000 件上限にする案ではいかがでしょうか？」 |
| 4 | 「DB は SQLite で十分でしょ」 | 4 妥当性に疑問 | 推論内容を返して議論 → 結論後に反映 | 「同時書き込みが想定 100 QPS のため SQLite だとロック競合のリスクがあります。Postgres を推奨しますが、要件次第で SQLite でも対応可能です」 |
| 5 | 「ここってどうなってる？」 | 1 質問 | 質問に回答のみ、本文は触らない | 「現状 `Service` クラスで処理しています。詳細は `モジュール構成_xxx.md` 参照」 |
