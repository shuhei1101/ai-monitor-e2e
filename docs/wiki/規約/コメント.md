# ai-monitor: コメント規約

## コメントの構造

1 つのコメント内で議論を続ける。
エージェント / ユーザーが交互に `---` 区切りでブロックを追記していく。
各ブロックの先頭には from 行（`> from: @{送信者}`）・to 行（`> to: @{宛先}`）の引用ヘッダーが付き、**from 行の有無が AI / 人間の判別になる**（エージェントは必須・ユーザーは書かない。to 省略 = その面の確認ラベルを保持している現担当エージェント宛）。
書式の組み立て・自分宛の判定・Resolve は MCP ツール（`comment` / `reply_comment` / `ask_questions` / `list_addressed_comments` / `resolve_comments`）が担う。

### ユーザー名（GitHub ログイン名）の取得方法

MCP の `receiver` にユーザー宛先を指定するときに使う。

```bash
gh api user --jq '.login'
```

## ユーザーコメントへの対応

ユーザーのコメントは「指示」「質問」「提案」「議論」など意図が様々。
ユーザーの言葉を鵜呑みにせず、AI が**自分で推論**してから対応する。

### コメントの分類と対応パターン

| ユーザーコメントの種類 | AI の対応 |
| --- | --- |
| 質問（「ここってどうなってる？」など） | 質問に答える追記のみ。本文は触らない |
| 明確かつ妥当な指示（AI も合意） | 即本文反映 → 「本文の `## XXX` を `Y` に修正しました」と短く追記 |
| 不明確な指示・複数解釈可能な指示 | ワンクッション: 「この理解で合っていますか？こう書き換えます」と確認の追記 → ユーザー合意後に反映 |
| 妥当性に疑問がある指示・提案 | 推論内容を返す: 「ご指摘の方法だと {懸念点}。代替案として {案} はいかがでしょうか？」 → 議論 → 結論が出たら反映 |
| 議論・相談（「これどう思う？」） | AI の意見を返す → 結論が出たら反映 |

### 共通ルール

- 他エージェントの報告を上位へ中継する場合（エスカレーション等）は、要約 + 元コメントへのリンクを添える
- 完了報告など**受領のみで完結するコメント**は処理時に Resolve する。
  **対応作業を求める報告**（差し戻し・レビュー指摘・失敗報告等）は、対応内容（修正 commit の ID 等）を**同スレッドに返信追記**して対応が確定してから Resolve する（どの経緯で発生した修正かをスレッドで追えるようにする）
- 本文反映後の応答は短く（「本文の `## XXX` を更新しました」程度）
- 本文反映時は変更前の本文記述は消すこと（常に「最新の確定版」に保つ。古い情報は残さない。）
- 部分書き換えは難しいので、各エージェント担当セクションを丸ごと置換
- コメントは個別に Resolve しない（ユーザーが反映結果を確認できなくなるため）。
  - Resolve は完了処理時（`議論中` 除去後）にまとめて行う

## Resolve のタイミング

ユーザーが `議論中` ラベルを外したタイミング（完了処理）で、AI が**自分宛のコメントを一括 Resolve** する。

### 精査対象（自分宛のコメント）

| コメント分類 | 自分宛か | 扱い |
| --- | --- | --- |
| 自身のエージェントが投稿したコメント | はい | 反映状況を確認 → 一括 Resolve 対象 |
| ユーザー → 自身のエージェント宛のコメント | はい | 反映状況を確認 → 一括 Resolve 対象 |
| ユーザーの宛先なしコメント（自分が現担当のとき） | はい | 反映状況を確認 → 一括 Resolve 対象 |
| 他のエージェントが投稿したコメント | いいえ | 無視（当該エージェントの完了処理時に Resolve される） |
| ユーザー → 他のエージェント宛のコメント | いいえ | 無視 |

宛先は最後のブロックの to 行で判定し、**to 行がないユーザーブロックは現担当（その面の確認ラベル保持エージェント）宛として扱う**。
確認ラベルが外れている間に投稿された宛先なしコメントは、次にその面の確認ラベルを持ったエージェントが拾う（毎ターン再同期）。

### フロー

1. ユーザーが `議論中` を外す
2. AI が起動 → 自分宛のコメント（精査対象表で自分宛のもの）だけを精査
3. 自分宛の未反映コメントがあるか確認:
   - 全て本文反映済み → 自分宛コメントを一括 Resolve → 次のラベルへ進める
   - 未反映あり → ユーザーに「これは反映してよいですか？反映不要ですか？」と質問コメント → `assignee=ユーザー` で待機 → ユーザー回答後に再判定
4. Resolve は MCP `resolve_comments` で実行

## 対応例

| ユーザーコメント | 分類 | AI の対応 | AI の追記内容 |
| --- | --- | --- | --- |
| 「ライブラリは LangChain ではなく Anthropic SDK にして」 | 明確な指示 | 本文「採用ライブラリ」を更新（コメントは残す） | 「採用ライブラリを Anthropic SDK に変更しました」 |
| 「分割案 B でお願い」 | 明確な指示 | 本文「設計方針」を更新 + 子 Issue 作成（コメントは残す） | 「分割案 B を採用、子 Issue #45/#46 を作成しました」 |
| 「キャッシュ層、もう少しシンプルにできない？」 | 不明確な指示 | ワンクッションで確認 → ユーザー合意後に反映 | 「インメモリ LRU に変更しキャッシュサイズを 1000 件上限にする案ではいかがでしょうか？」 |
| 「DB は SQLite で十分でしょ」 | 妥当性に疑問 | 推論内容を返して議論 → 結論後に反映 | 「同時書き込みが想定 100 QPS のため SQLite だとロック競合のリスクがあります。Postgres を推奨しますが、要件次第で SQLite でも対応可能です」 |
| 「ここってどうなってる？」 | 質問 | 質問に回答のみ、本文は触らない | 「現状 `Service` クラスで処理しています。詳細は `モジュール構成_xxx.md` 参照」 |
