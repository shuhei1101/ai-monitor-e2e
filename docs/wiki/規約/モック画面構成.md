# ai-monitor: モック画面構成規約

モック画面の置き場所と命名の規約。
プロジェクトの feature ブランチで `docs/mock/` 配下に作成し、master マージ後は GitHub Pages（Branch: `master`、Folder: `/docs`）で配信する。
開発中のモック確認は `raw.githack.com` で feature ブランチの HTML を直接描画して見る（後述「配信方針」参照）。

新規モック画面を作るときは「ディレクトリ構成」のツリーに沿って新しいディレクトリを切り、上位の索引（`docs/mock/pages/index.md` または `docs/mock/components/index.md`）にリンクを 1 行追加する。

## 配信方針

| 状態 | 配信方法 | URL 例 |
| --- | --- | --- |
| 開発中（feature ブランチ） | raw.githack.com（CDN で raw HTML を描画） | `https://raw.githack.com/shuhei1101/my-plugins/{ブランチ名}/docs/mock/pages/{画面名}/issues/{N}/{案名}/index.html` |
| 確定後（master マージ後） | GitHub Pages（Jekyll ビルド） | `https://shuhei1101.github.io/my-plugins/mock/pages/{画面名}/issues/{N}/{案名}/` |

### raw.githack.com の使い方

- mock-designer フェーズでモック HTML を作成 → feature ブランチに push
- raw.githack.com の URL でブラウザから直接描画 → Issue / PR コメントに URL を貼る
- 外部 CSS / JS 参照（`assets/style.css` 等）も同じ raw.githack.com 経由で解決される
- ブランチごとに独立 URL になるため、PR ごとに別モックを並行検討できる
- 索引などの Markdown ページは Jekyll ビルド前提のため、raw.githack.com では見られない。
  索引の確認は GitHub の Web UI（MD レンダリング）または GitHub Pages で行う

## 配信の仕組み（Jekyll）

`docs/` 配下は GitHub Pages が Jekyll でビルドする。
Markdown は自動で HTML に変換され、`docs/_layouts/default.html` の器に流し込まれる。
つまり `index.md` を置くだけでフォルダ URL がそのまま HTML ページとして開ける（`index.html` シェルを毎回作る必要はない）。

| ファイル | 役割 |
| --- | --- |
| `docs/_config.yml` | Jekyll 設定。`jekyll-optional-front-matter` + `jekyll-readme-index` + `defaults` で MD 側ゼロタッチ |
| `docs/_layouts/default.html` | 全ページ共通の器（topbar / パンくず / 本文 / 戻るボタン）。1 ファイルのみ |
| `docs/assets/style.css` | 全ページ共通スタイル |
| `docs/assets/app.js` | 全ページ共通スクリプト（パンくず / 表ソート / 表フィルタ / 戻るボタン / Mock demo 用ハンドラ） |

MD 側に `---\nlayout: default\n---` を書く必要はない（`_config.yml` の `defaults` が自動適用する）。

## ディレクトリ構成

```
docs/
├── _config.yml                                         # Jekyll 設定
├── _layouts/
│   └── default.html                                    # 全ページ共通の器
├── assets/                                             # docs 全体の共通アセット
│   ├── style.css                                       # 共通スタイル（コメント付き）
│   └── app.js                                          # 共通スクリプト（1 ファイル集約）
├── index.md                                            # サイトトップ
└── mock/
    ├── index.md                                        # モック索引（pages / components / styles / issues への入口）
    ├── pages/
    │   ├── index.md                                    # 画面索引（画面リスト + 現役採用案リンク）
    │   └── {画面名}/
    │       ├── index.md                                # Issue 一覧（最新順）
    │       └── issues/
    │           └── {issue番号}/
    │               ├── index.md                        # 当該 Issue の案リスト（採用マーク付き）
    │               └── {案名}/
    │                   ├── index.html                  # 案の実画面（HTML のまま。MD 化しない）
    │                   └── {派生案名}/                 # 任意。派生案はネスト可
    │                       └── index.html              # 派生案の実画面
    ├── components/
    │   ├── index.md                                    # コンポーネント索引
    │   └── {コンポーネント名}/
    │       └── index.html                              # 全案をまとめたカタログ（HTML のまま）
    ├── styles/
    │   ├── index.md                                    # スタイル一覧（採用 + 候補）
    │   └── {スタイル名}/
    │       └── index.html                              # 当該スタイルのサンプル画面（HTML のまま）
    └── issues/
        ├── index.md                                    # Issue 索引（最新順）
        └── {issue番号}/
            └── index.md                                # 当該 Issue のリンク集
```

`{画面名}` `{案名}` `{派生案名}` `{コンポーネント名}` `{スタイル名}` は kebab-case。
例: `customer-list`、`a-left-filter`、`with-bulk-edit`、`topbar`、`modern-light`。

`{issue番号}` は GitHub Issue 番号をそのまま使う（例: `50`、`123`）。
`pages/{画面名}/50/` と `issues/50/` が同じ番号で対応し、相互に辿りやすくなる。

画面とコンポーネントで階層方針が異なる:

- 画面: Issue ごとに案を分ける（`pages/{画面名}/issues/{issue番号}/{案名}/`）
  - 同じ画面でも新しい Issue で改修が入ると新しい `issues/{issue番号}/` を切り、当該 Issue で複数案を検討・採用する
- コンポーネント: 案を別ページに分けず、`components/{コンポーネント名}/index.html` 1 ページに全案を並べる
  - ボタンやサイドバーのように単体の見た目で比較できるものが対象で、Issue 単位の改修概念も持たない

## Markdown ページと HTML ページの使い分け

| ページ種別 | 実体 | 補足 |
| --- | --- | --- |
| 索引・案リスト・Issue リストなど「読ませる」ページ | `index.md` | Jekyll が `default.html` の器に流し込む |
| 案の実画面（`pages/.../{案名}/index.html`） | 実画面 HTML | Mock として見せる画面本体。Jekyll の器は使わず単独 HTML |
| コンポーネントカタログ（`components/{名}/index.html`） | 実画面 HTML | カタログを 1 画面で見せるため MD 化しない |
| スタイルサンプル（`styles/{名}/index.html`） | 実画面 HTML | スタイルサンプル画面のため MD 化しない |

## 各階層 `index.*` の中身

| 階層 | `index.*` の中身 | 補足 |
| --- | --- | --- |
| `pages/` | 画面索引（`index.md`） | - |
| `pages/{画面名}/` | Issue 一覧（`index.md`） | - |
| `pages/{画面名}/issues/{issue番号}/` | 案リスト（`index.md`） | - |
| `pages/{画面名}/issues/{issue番号}/{案名}/` | 案の実画面（`index.html`） | - |
| `pages/{画面名}/issues/{issue番号}/{案名}/{派生案名}/...` | 派生案の実画面（`index.html`）。さらにネスト可能 | - |
| `components/` | コンポーネント索引（`index.md`） | - |
| `components/{コンポーネント名}/` | 全案カタログ（`index.html`） | - |
| `styles/` | スタイル一覧（`index.md`） | - |
| `styles/{スタイル名}/` | 当該スタイルのサンプル画面（`index.html`） | - |
| `issues/` | Issue 索引（`index.md`） | - |
| `issues/{issue番号}/` | 当該 Issue のリンク集（`index.md`） | - |

## 採用スタイルと共通アセットの関係

採用が決まったスタイルは、共通アセット `docs/assets/style.css` の内容と一致させる。
共通アセットを「採用スタイル本体」とし、`styles/{採用スタイル名}/` は「採用スタイルのサンプル展示」として残す。
未採用の候補スタイルは `styles/{スタイル名}/style.css` でそれぞれ独自の見た目を表現する。

リンク方針:

- 改修ごとに 1 リンクで足りる場合は案リスト（`pages/{画面名}/issues/{issue番号}/`）を指す

- 1 Issue で扱った全画面を漏らさず載せる

## 画面 UI 構成（全 `index.html` 共通）

モック実画面（`pages/.../{案名}/index.html` を中心とした各 `index.html`）の画面内の UI 要素に関する規約。
ディレクトリ構成と直交するルールで、全モック画面に適用する。

### トップバー

各画面の最上部に置く帯。
以下のルールに従う。

| 要素 | 扱い | 補足 |
| --- | --- | --- |
| サイト名 / ロゴ | 任意 | 置く場合は左端。クリックで `docs/index.md`（または `docs/mock/index.md`）へ戻る |
| 同階層の案タブ | 必須（案複数時） | 同じ `pages/{画面名}/issues/{issue番号}/` 配下の `{案名}` をタブで横並び |

#### 同階層の案タブ

- 現在開いている `{案名}` と兄弟の `{案名}` をタブとして横並びに表示する（例: `案A` `案B` `案C`）
- リンク先は各案の `index.html`（相対パス: `../{案名}/index.html`）
- 現在開いている案は active 状態で強調表示（背景色 / 下線など）
- 派生案（`{派生案名}/`）配下の画面では、親 `{案名}` の兄弟タブ + 自身の派生案兄弟タブの 2 段で並べてよい
- タブが収まり切らない場合は横スクロール対応（`overflow-x: auto` + タッチ操作可）。
  折り返しはしない
- 案が 1 つだけのときはタブそのものを表示しない（タブは比較目的のため）

### パンくずリスト

- トップバーではなく画面本体（ページコンテンツの最上部）に置く
- Markdown ページは Jekyll の `_layouts/default.html` に組み込まれた `<nav data-breadcrumb>` を共通 `app.js` が URL 階層から自動生成する。
  MD 側で書く必要はない
- 実 mock 画面（`{案名}/index.html`）は自前でパンくずを書く。
  以下のルールに従う
  - 末端の `{案名}` まで開いている場合、`pages` から末端まで全階層を遡れるリンク列にする
  - 例: `pages / customer-list / issues / 50 / a-left-filter / with-bulk-edit`
  - 各セグメントは該当ディレクトリの `index.html` を指すリンク
  - 末端セグメント（現在地）はリンク化せず通常テキスト表示
  - セパレーターは `/` または `›` で統一

### 採用マーク

- 案リスト・派生案・スタイル一覧などで採用案を識別するためのバッジ
- 共通スタイル `assets/style.css` に `.badge.ok` などのクラスを定義し、各画面はクラスを当てるだけで使えるようにする
- 文言は短く（例: `採用` / `OK` / `Adopted`）

### 共通アセットの参照

共通アセットは `docs/assets/` に置く。

- Markdown ページ: `_layouts/default.html` が Jekyll の `relative_url` フィルタで自動解決する（MD 側で書く必要なし）
- 実画面 HTML（`{案名}/index.html` 等）: ネスト階層に応じて `../` を積む

実画面 HTML の相対パス目安:

| ファイル位置 | 共通アセットへの相対パス |
| --- | --- |
| `docs/mock/pages/{画面名}/issues/{issue番号}/{案名}/index.html` | `../../../../../../assets/` |
| `docs/mock/pages/{画面名}/issues/{issue番号}/{案名}/{派生案名}/index.html` | `../../../../../../../assets/` |
| `docs/mock/components/{コンポーネント名}/index.html` | `../../../../assets/` |
| `docs/mock/styles/{スタイル名}/index.html` | `../../../assets/` |

パスがずれると共通スタイルが当たらず崩れるため、新規画面追加時はテンプレートを必ずコピーする。
