# 分解判定（初回）

intake Issue の内容を作業単位に分解して種別を判定し、サブ Issue 案 + 確認質問を投稿して待機に入る。

## 手順

### 関連情報の調査

本文から機能名・画面名・エラーメッセージなどのキーワードを抽出し、分解と確認質問の材料になる既存情報を集める。

- 関連 Issue / PR: MCP `search_issues_and_prs` をキーワードごとに呼び（`query`: キーワード。`is:issue` / `is:pr` 等の修飾子可）、依存・重複しそうな番号と概要を控える
- 既存設計: 注入済みのシナリオ索引（`設計図/シナリオ/README.md`）から関連する単一 / 複合ユースケースを選び、オンデマンド取得で読む（手順は共通ルール『Wikiページのオンデマンド取得』）

### 作業単位への分解

Issue の本文を読み、含まれる作業を独立したスコープごとに分解する。

- 依存関係 or 共通目的あり → 1 サブ Issue にまとめる
- 独立した機能 / 目的が混在 → 複数サブ Issue に分割
- 迷ったら「同じ PR ツリーで進行するか」で判断

### 種別の判定

各作業について、判定フローチャート（レイヤー）に沿って種別を割り当てる。
種別ごとにサブ Issue へ付与するラベル（値は環境変数を `echo` で解決する）:

| 種別 | layer ラベル | 確認ラベル |
| --- | --- | --- |
| epic | `$AI_MONITOR_LABEL_LAYER_EPIC` の値 | `$AI_MONITOR_LABEL_CONFIRM_EPIC_CONDUCTOR` の値 |
| story | `$AI_MONITOR_LABEL_LAYER_STORY` の値 | `$AI_MONITOR_LABEL_CONFIRM_STORY_CONDUCTOR` の値 |
| subsystem | `$AI_MONITOR_LABEL_LAYER_SUBSYSTEM` の値 | `$AI_MONITOR_LABEL_CONFIRM_SUBSYSTEM_CONDUCTOR` の値 |
| chore | `$AI_MONITOR_LABEL_LAYER_CHORE` の値 | `$AI_MONITOR_LABEL_CONFIRM_QUICK_IMPLEMENTER` の値 |

### type / layer ラベルの付与

intake Issue には集約元を示す layer ラベルと、内容から判定した type ラベルを付ける。

type ラベルは以下から 1 つ判定する:

- 既存の動作が仕様または期待と異なる問題の修正: `$AI_MONITOR_LABEL_TYPE_BUG` の値
- 新機能追加・既存機能の有意な拡張: `$AI_MONITOR_LABEL_TYPE_FEAT` の値
- 外部動作を変えずにコードを整理・改善: `$AI_MONITOR_LABEL_TYPE_REFACTOR` の値
- ドキュメント・コメントのみの変更: `$AI_MONITOR_LABEL_TYPE_DOCS` の値
- ビルド設定・依存更新・CI/CD など: `$AI_MONITOR_LABEL_TYPE_CHORE` の値
- テストコードの追加・修正のみ: `$AI_MONITOR_LABEL_TYPE_TEST` の値

MCP `add_labels` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `labels`:
  - `$AI_MONITOR_LABEL_LAYER_INTAKE` の値
  - 判定した type ラベルの値

### サブIssue案の提示

判定結果を全件まとめてユーザーに提示する。

MCP `comment` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `sender`: `intake-issue-triager`
- `receiver`: ユーザーログイン名（`gh api user --jq '.login'` で取得）
- `body`:
  ```
  Issue の内容を以下の通り分解しました。

  | 種別 | タイトル案 | 概要 |
  | --- | --- | --- |
  | {epic/story/subsystem/chore} | {タイトル案} | {概要} |

  - 問題なければ `議論中` ラベルを外す → 承認された案で Sub-issue として起票
  - 修正が必要なら assignee を外してフィードバックコメントを記入
  ```

### 確認質問の投稿

以下の観点から今回の Issue に該当するものだけを抽出し、AI 側で推奨案を用意して投稿する。

| 観点 | 質問 | 選択肢例 | 推奨 |
| --- | --- | --- | --- |
| 要件間の整合性 | 「A と B が矛盾するように読めます。どちらを優先しますか？」 | A. {要件A を優先} / B. {要件B を優先} / C. 両立させる方法を検討 | A |
| スコープ境界 | 「〜も含めますか？」 | A. 含めない（今回スコープ外） / B. 含める | A |
| 既存機能との関係 | 「既存の {類似機能} との扱いは？」 | A. 統合する / B. 別立てで併存 / C. 既存を廃止して置き換え | B |
| 廃止・置換の意図 | 「既存の {既存機能} は？」 | A. 併存 / B. 段階的廃止（新規は新機能、既存はそのまま） / C. 即時廃止 | A |
| 優先順序 / 完了順 | 「複数作業のどれから完了させたいですか？」 | A. {作業 X から} / B. {作業 Y から} / C. 並行で進める | A |
| 前提条件の妥当性 | 「「〜がある前提」ですが、その前提は満たされていますか？」 | A. 満たされている / B. 満たされていない（先に {対応} が必要） | A |
| 依存 Issue / PR の確認（調査でヒットした場合） | 「#{番号} と依存 / 重複がありそうです。関係の理解は合っていますか？」 | A. 合っている / B. 無関係 / C. 補足あり（{内容}） | A |
| バグ再現条件（type:bug 時） | 「再現手順・環境の理解は合っていますか？」 | A. 合っている / B. 追加条件がある（{条件}） | A |
| 影響ユーザー / アクター | 「影響を受けるユーザー範囲は？」 | A. 全ユーザー / B. 特定ロール（{ロール}）のみ / C. 一部条件（{条件}）に該当するユーザー | A |
| 想定タイムライン | 「いつまでに完了させたい想定ですか？」 | A. 特に指定なし / B. 今週中 / C. 今月中 / D. 次リリース | A |
| 非機能要件の存在 | 「セキュリティ / 性能 / 対応環境 で気にすべきことはありますか？」 | A. なし（デフォルト方針で OK） / B. あり（{内容}） | A |
| 用語の意味 | 「本文の「{用語}」は {案 X} の意味ですか？」 | A. {案 X} で合っている / B. {案 Y} が正しい | A |
| 対応方針の代替案 | 「対応方針の候補は？」 | A. {案 A の要約} / B. {案 B の要約} / C. その他 | A |

MCP `ask_questions` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `sender`: `intake-issue-triager`
- `receiver`: ユーザーログイン名
- `intro`: `以下の点についてご確認ください。記号（例: A）で返信いただければ、その方針で進めます。`
- `questions`: 上記観点表から抽出

### 議論中 付与 + 待機

MCP `add_labels` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `labels`:
  - `$AI_MONITOR_LABEL_IN_DISCUSSION` の値

続けて MCP `set_assignee` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
