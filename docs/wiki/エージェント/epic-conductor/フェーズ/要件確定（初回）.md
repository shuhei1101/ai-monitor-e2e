# 要件確定（初回）

epic Issue 本文（前提条件 / 概要 / 背景 / ユースケース一覧 / 横断要件）の草案を作成し、実現可能性 PoC の要否と画面変更の有無を質問して待機に入る。

## 手順

### 親 intake の確認

MCP `get_issue_or_pr` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `parent`: true

親 intake の本文と承認済みのサブ Issue 案から、この epic が担当する範囲を把握する。

### 本文の作成

テンプレート（イシュー本文/エピック）に沿って 5 セクション（`## 前提条件` / `## 概要` / `## 背景` / `## ユースケース一覧` / `## 横断要件`）の草案を作成する。

- `## 概要` / `## 背景` はユーザー入力（親 intake の記述）の範囲内で整文する
- `## ユースケース一覧` の `対応 story` 列は全行 `未起票` にする

MCP `update_body` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `body`: 作成した本文

### 完了報告の投稿

MCP `comment` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `sender`: `epic-conductor`
- `receiver`: ユーザーログイン名（`gh api user --jq '.login'` で取得）
- `body`: 本文草案の完了報告（確認してほしい点の要約 + 承認・修正方法の案内）

### 確認質問の投稿

以下の 2 問は必ず含め、内容に応じた確認事項（ユースケースの過不足・横断要件の水準など）があれば加える。

| 観点 | 質問 | 選択肢例 | 推奨 |
| --- | --- | --- | --- |
| 実現可能性 PoC 要否 | 「この epic の成立は前例のない技術機構（未検証のプロトコル連携・性能が成立条件 等）に依存しますか？」 | A. 依存しない（PoC 不要） / B. 依存する（PoC 必要: {検証テーマ}） | A |
| 画面変更の有無 | 「画面の新規作成・レイアウト変更を含みますか？」 | A. 含まない / B. 含む | A |

MCP `ask_questions` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `sender`: `epic-conductor`
- `receiver`: ユーザーログイン名
- `intro`: `以下の点についてご確認ください。記号（例: A）で返信いただければ、その方針で進めます。`
- `questions`: 上記の質問

### 議論中 付与 + 待機

MCP `add_labels` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `labels`:
  - `$AI_MONITOR_LABEL_IN_DISCUSSION` の値

続けて MCP `set_assignee` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false

### 作業完了報告

MCP `report_completion` を呼ぶ:
- `agent_name`: `epic-conductor`
- `number`: $issue_number
