# 統合テスト起動

story-conductor の完了報告（story 完了）を受けて全子 story の close 状態を確認し、全完了なら複合 UC 統合テスト一式を complex-scenario-writer に委任する。
統合テストの内部工程（実装 → レビュー → 実行）は writer が指揮する。

## 手順

### 全子 story の確認

初期処理で取得した Sub-issue の state を確認する。

### 完了報告の Resolve と分岐

MCP `resolve_comments` で story-conductor の完了報告コメントを Resolve する。

- 全子 story が closed の場合、MCP `add_labels` を呼ぶ:
  - `number`: epic PR の番号
  - `is_pr`: true
  - `labels`:
    - `$AI_MONITOR_LABEL_CONFIRM_COMPLEX_SCENARIO_WRITER` の値（統合テストの委任。初回実装か再テストかの判定は writer が行う）
- open の story が残っている場合、open の各 story Issue に状況確認を送る（残りの story の完了報告で再復帰する）:
  1. MCP `comment` を呼ぶ（`number`: open の story Issue 番号・`is_pr`: false・`sender`: `epic-conductor`・`receiver`: `story-conductor`・`body`: 進捗とブロッカーの確認）
  2. MCP `add_labels` を呼ぶ（`number`: open の story Issue 番号・`is_pr`: false・`labels`: `$AI_MONITOR_LABEL_CONFIRM_STORY_CONDUCTOR` の値）

### ラベル除去

MCP `transition_phase` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `remove_labels_`:
  - `$AI_MONITOR_LABEL_CONFIRM_EPIC_CONDUCTOR` の値
- `add_labels_`: なし（ユーザー承認なしの自動完了）

### 作業完了報告

MCP `report_completion` を呼ぶ:
- `agent_name`: `epic-conductor`
- `number`: $issue_number
