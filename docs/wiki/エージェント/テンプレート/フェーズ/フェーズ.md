# フェーズ

## セクション一覧

| セクション | 必須or条件 |
| --- | --- |
| 冒頭リード | 必須 |
| `## 手順` | 必須 |

## `冒頭リード`

### 記述例

```markdown
# 要件確定（初回）

story Issue 本文の骨組みを作成し、ユースケース要件の草案 + 確認質問を投稿して待機に入る。
```

### 補足

- 1 行目はフェーズ名（フェーズ索引の行・ファイル名と一致）
- このフェーズで達成することを 1〜2 行で書く（起動条件はフェーズ索引側に書く）

## `## 手順`

### 記述例

```markdown
## 手順

### 親 Issue の本文を取得

MCP `get_issue_or_pr` を呼ぶ:
- `number`: 初期処理で取得した `parent.number`
- `is_pr`: false

### 本文の骨組み作成

イシュー本文テンプレートの構成に沿って各セクションの草案を作る。

MCP `update_body` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `body`: 草案を反映した本文

### 自分宛コメントの選別

- 全て本文反映済みの場合、MCP `resolve_comments` で一括 Resolve する
- 未反映がある場合、確認質問を投稿して待機に入る（「議論中 付与 + 待機」は実行しない）

### 議論中 付与 + 待機

MCP `add_labels` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `labels`: `$AI_MONITOR_LABEL_IN_DISCUSSION` の値

続けて MCP `set_assignee` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
```

### 補足

- 1 手順 = 1 H3 セクション（`### {手順名}`）で分け、上から順に実行する
- 手順名は内容の日本語名で書き、説明が必要なら見出し直下に 1〜2 行で書く
- MCP ツールの呼び出しは `MCP `{ツール名}` を呼ぶ:` + 引数の箇条書きで書く（連続して呼ぶ場合は `続けて MCP `{ツール名}` を呼ぶ:`）
- 分岐は `- {条件}の場合、{内容}` の箇条書きで並べる（後続手順のスキップは `（「{手順名}」は実行しない）` と手順名で明記）
- ラベル値は環境変数名（`$AI_MONITOR_LABEL_*`。`constants.env` 由来で SessionStart フックが展開済み）で指定し、実行時に `echo` で解決する
- フェーズの終端は 待機（`議論中` 付与 + `set_assignee`）か 完了（`transition_phase` で `確認:{自身}` 除去 + 次の `確認:*` 付与）のどちらかで終える
- 終端の直後に `### 作業完了報告` を置き、MCP `report_completion`（`agent_name` + 主番号）でモニターへターン終了を通知する（`処理中:*` の除去に必要）
- このフェーズでしか使わない Wiki ページは手順内で実行時取得する（`python "${CLAUDE_PLUGIN_ROOT}/inject/read_urls.py" "{URL}"`）

### 記述例（初期処理）

全エージェント共通で先頭に置くフェーズの記述例:

```markdown
# 初期処理

毎ターン（初回・応答ループ・完了処理のどの呼び出しでも）最初に実行する。

## 手順

### 最新状態の取得

MCP `get_issue_or_pr` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false

続けて MCP `list_addressed_comments` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `addressee`: `story-conductor`
- `include_resolved`: false

### フェーズの決定

フェーズ索引のフェーズ一覧を上から順に見て、起動条件に最初にマッチしたフェーズの手順を実行する。
前ターンの記憶ではなく取得結果を判定材料にする（本文・ラベル・コメントはターン間にユーザーや他エージェントが変更している可能性がある）。
```
