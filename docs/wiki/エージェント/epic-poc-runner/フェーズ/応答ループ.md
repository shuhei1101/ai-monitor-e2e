# 応答ループ

待機中のユーザーフィードバックに対応し、再び待機に入る。
どの待機（方針固め・結果報告）かは、PoC PR 本文の状態（`## 検証結果` の記入有無）と未解決の自分宛スレッドの文脈で判断する。

## 手順

### 内容に応じた対応

| 待機の文脈 | 対応 |
| --- | --- |
| 方針固めの本文修正指示 | 本文の該当セクション（`## リスク仮説` / `## 検証構成` / `## 成功条件`）を修正（`update_body`）して「本文の `## XXX` を更新しました」と短く追記 |
| 方針固めの質問への回答 | 回答を本文へ反映（草案の確定要素として `update_body`）して短く追記 |
| 結果報告への追加検証指示 | 「検証実行」フェーズを再度回すため、本文の `## 検証結果` / `## 最小再現コード` を空にして（`update_body`）「追加検証を実行します」と追記（次ターンで検証実行フェーズが再起動する） |
| 代替案の指示 | 本文の `## 検証構成` / `## 成功条件` を代替案に書き換え + `## 検証結果` を空に（`update_body`）して短く追記 |
| 中止の指示 | 「中止する場合は PoC PR に `確認:resetter` ラベルを付与してください」と案内を短く追記（本文・ラベルは触らない。`確認:resetter` の付与は唯一のユーザー手動起動エージェントで、AI は付与しない） |

追記は MCP `reply_comment` を呼ぶ:
- `comment_node_id`: 返信対象コメントの node_id
- `sender`: `epic-poc-runner`
- `receiver`: ユーザーログイン名
- `body`: 返信本文

### assignee 再設定

MCP `set_assignee` を呼ぶ:
- `number`: $pr_number
- `is_pr`: true

### 作業完了報告

MCP `report_completion` を呼ぶ:
- `agent_name`: `epic-poc-runner`
- `number`: $pr_number
