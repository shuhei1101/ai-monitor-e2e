# 方針固め（初回）

epic-conductor の指示コメントと親 epic Issue を確認し、PoC PR 本文にリスク仮説 / 検証構成 / 成功条件の草案を書いて、不明点を質問して待機に入る。

## 手順

### 指示コメントと親 epic Issue の確認

初期処理で取得した未解決の自分宛コメントから epic-conductor の指示コメント（検証テーマの背景 + 成立条件の想定）を読む。

続けて親 epic Issue の内容を取得する。
PoC PR 本文の `## 紐づく Issue` から親 epic Issue 番号を得て MCP `get_issue_or_pr` を呼ぶ:
- `number`: 親 epic Issue 番号
- `is_pr`: false
- `comments`: false

epic Issue 本文の `## 概要` / `## 横断要件` から仮説の元ネタを抽出する。

### 本文の仮埋め

テンプレート（PR本文/エピックPoC）に沿って 3 セクション（`## リスク仮説` / `## 検証構成` / `## 成功条件`）の草案を作成する。

- `## リスク仮説`: 「{機構} が成立しなければ {epic への影響}」の形で 1〜3 文
- `## 検証構成`: 最安直構成の Mermaid 図 + 使用ライブラリ表（銘柄比較はしない・代表選定）
- `## 成功条件`: 数値 or 観測可能な基準の表

MCP `update_body` を呼ぶ:
- `number`: $pr_number
- `is_pr`: true
- `body`: `## 紐づく Issue` + 作成した 3 セクション

### 完了報告の投稿

MCP `comment` を呼ぶ:
- `number`: $pr_number
- `is_pr`: true
- `sender`: `epic-poc-runner`
- `receiver`: ユーザーログイン名（`gh api user --jq '.login'` で取得）
- `body`: 本文草案の完了報告（確認してほしい点の要約 + 承認・修正方法の案内）

### 確認質問の投稿

不明点があれば確認質問を投げる（無ければスキップして「議論中 付与 + 待機」へ進む）。

以下の観点から今回の PoC に該当するものだけを抽出し、指示コメント + 親 epic Issue の内容から AI 側で推奨案を判断して投稿する。

| 観点 | 質問 | 選択肢例 |
| --- | --- | --- |
| 検証構成の妥当性 | 「最安直構成としてこの構成で検証してよいですか？」 | A. この構成で OK / B. 別の構成（{案}）を推奨 |
| 成功条件の水準 | 「{条件}の基準値は {値} で十分ですか？」 | A. この値で OK / B. より厳しく（{値}）/ C. より緩く（{値}） |
| 追加案の要否 | 「別方式（{代替案}）も並列で検証しますか？」 | A. 不要（この 1 案でよい） / B. 必要（{代替案}を追加） |
| 検証対象の範囲 | 「{周辺機能}も検証対象に含めますか？」 | A. 含めない（今回はスコープ外） / B. 含める |

MCP `ask_questions` を呼ぶ:
- `number`: $pr_number
- `is_pr`: true
- `sender`: `epic-poc-runner`
- `receiver`: ユーザーログイン名
- `intro`: `以下の点についてご確認ください。記号（例: A）で返信いただければ、その方針で進めます。`
- `questions`: 上記観点表から抽出

### 議論中 付与 + 待機

MCP `add_labels` を呼ぶ:
- `number`: $pr_number
- `is_pr`: true
- `labels`:
  - `$AI_MONITOR_LABEL_IN_DISCUSSION` の値

続けて MCP `set_assignee` を呼ぶ:
- `number`: $pr_number
- `is_pr`: true

### 作業完了報告

MCP `report_completion` を呼ぶ:
- `agent_name`: `epic-poc-runner`
- `number`: $pr_number
