# ライブラリ選定

architect が設計の応答ループ中に発生したライブラリ選定論点について、候補比較 → （必要なら PoC 発注）→ 採用決定 → 外部ライブラリ Wiki 反映を行う単一ユースケース。
候補ごとの PoC 検証は `library-poc-runner` へ発注する（1 候補 = 1 PoC PR = 1 セッション並列）。
BE ライブラリ / UI ライブラリで共通のフロー。

対応エージェント: `architect`

## 正常シナリオ（PoC あり）

### セットアップ

| セットアップ | 説明 | 補足 |
| --- | --- | --- |
| Mock | なし（実環境で実行） | - |
| subsystem PR | 設計 Wiki の応答ループ中（担当 worker の確認ラベル + `議論中` あり） | - |
| ライブラリ選定論点 | 応答ループ中に発生 | 例: LLM クライアント / UI コンポーネントライブラリの採用 |
| 採用候補 | 未経験のライブラリで PoC 要否判定カテゴリ（A〜E）に該当 | PoC を誘発 |

### フロー

```mermaid
sequenceDiagram
  actor U as ユーザー
  participant GH as GitHub
  participant ORC as モニター
  participant MON as architect
  participant REPO as リポジトリ

  Note over MON: 設計の応答ループ中に<br>ライブラリ選定論点が発生
  activate MON
  MON->>MON: 候補列挙 + 候補ごとの調査<br>（library-finder / library-researcher 並列）
  alt PoC 不要（要否判定カテゴリ非該当）
    Note over MON: 正常シナリオ<br>（PoC なしで採用決定）参照
  end
  MON->>GH: subsystem PR に候補比較 +<br>検証観点の提案コメント（スレッド起点）+<br>議論中 付与 + assignee=ユーザー 設定
  deactivate MON

  loop 応答ループ（候補・観点への修正指示がある間）
    U->>GH: subsystem PR にフィードバックコメント +<br>assignee 外し
    ORC-->>GH: polling（ユーザー返信 + assignee なし を検知）
    ORC->>MON: 既存セッションへ送信
    activate MON
    MON->>GH: 候補・検証観点を更新 +<br>assignee=ユーザー 再設定
    deactivate MON
  end

  U->>GH: 候補・検証観点に合意コメント +<br>assignee 外し
  ORC-->>GH: polling（ユーザー返信 + assignee なし を検知）
  ORC->>MON: 既存セッションへ送信
  activate MON
  MON->>REPO: 候補ごとに PoC worktree 作成 +<br>空 commit push<br>（poc/{scope}/{ドメイン}/{UC名}/{lib名}）
  MON->>GH: 候補ごとに PoC Draft PR 作成<br>（base=master・本文に検証対象 /<br>調査結果 / 検証観点を記載）
  MON->>GH: 候補比較コメントのスレッドに<br>PoC PR のリンク一覧を追記
  MON->>GH: 各 PoC PR に<br>確認:library-poc-runner 付与 +<br>検証指示コメント（@library-poc-runner 宛）
  MON->>ORC: 全 PoC PR の番号を<br>自セッションの監視面として台帳に登録
  deactivate MON

  Note over GH: 候補ごとの検証が並列で走り、<br>各 PoC PR に 確認:architect +<br>完了報告コメントが投稿される
  ORC-->>GH: polling（確認ラベル + 未解決コメント を検知）
  ORC->>MON: 既存セッションへ送信<br>（PoC PR 側は台帳の監視面から<br>architect のセッションへ届く）
  activate MON
  MON->>GH: 完了報告コメントを Resolve
  MON->>GH: subsystem PR に全候補の<br>結果まとめコメント（PoC PR リンク付き）+<br>議論中 付与 + assignee=ユーザー 設定
  deactivate MON

  loop 応答ループ（質問・追加検証の依頼がある間）
    U->>GH: subsystem PR に質問・追加検証の<br>依頼コメント + assignee 外し
    ORC-->>GH: polling（ユーザー返信 + assignee なし を検知）
    ORC->>MON: 既存セッションへ送信
    activate MON
    alt 追加検証が必要
      Note over MON: 正常シナリオ<br>（追加検証の再発注）参照
    end
    MON->>GH: subsystem PR で回答 +<br>assignee=ユーザー 再設定
    deactivate MON
  end

  U->>GH: 採用ライブラリの決定コメント +<br>assignee 外し
  ORC-->>GH: polling（ユーザー返信 + assignee なし を検知）
  ORC->>MON: 既存セッションへ送信
  activate MON
  MON->>REPO: 外部ライブラリ/README.md に行追加 +<br>外部ライブラリ/{lib名}.md 作成 commit push
  MON->>GH: subsystem PR に Wiki 反映の報告コメント +<br>議論中 付与 + assignee=ユーザー 設定
  deactivate MON

  loop 応答ループ（Wiki 反映への修正指示がある間）
    U->>GH: subsystem PR にフィードバックコメント +<br>assignee 外し
    ORC-->>GH: polling（ユーザー返信 + assignee なし を検知）
    ORC->>MON: 既存セッションへ送信
    activate MON
    MON->>REPO: Wiki 修正 commit push
    MON->>GH: subsystem PR の<br>assignee=ユーザー 再設定
    deactivate MON
  end

  U->>GH: 外部ライブラリ Wiki の承認コメント +<br>assignee 外し
  ORC-->>GH: polling（ユーザー返信 + assignee なし を検知）
  ORC->>MON: 既存セッションへ送信
  activate MON
  MON->>GH: subsystem PR の PoC 関連の<br>自分宛コメント一括 Resolve
  MON->>GH: 全 PoC PR を close<br>（マージなし・恒久記録）
  MON->>ORC: 全 PoC PR の番号を<br>監視面から除去
  MON->>REPO: PoC worktree と<br>PoC ブランチを全て削除
  MON->>GH: subsystem PR の<br>assignee=ユーザー 再設定
  deactivate MON
  Note over MON: 設計の応答ループに合流
```

### 期待値

- 候補ごとに PoC PR（base=master・本文に検証対象 / 調査結果 / 検証観点入り・`確認:library-poc-runner` + 検証指示コメント付き）が作成されている
- 発注時点で architect セッションの監視面（モニターの台帳）に全 PoC PR の番号が登録され、close 後の完了処理で除去されている
- 候補比較コメントのスレッドに全 PoC PR のリンク一覧が残っている
- `外部ライブラリ/README.md` の行と `外部ライブラリ/{lib名}.md` が subsystem ブランチに commit されている
- 全 PoC PR が closed（マージなし）で残り、PoC worktree・PoC ブランチ（ローカル / リモートとも）が削除済み
- 候補比較・検証結果・採用判断の経緯がコメントに残っている
- PoC 関連の自分宛コメントが全て Resolve 済み

## 正常シナリオ（追加検証の再発注）

### セットアップ

| セットアップ | 説明 | 補足 |
| --- | --- | --- |
| Mock | なし（実環境で実行） | - |
| 結果まとめまで完了 | 全候補の結果まとめコメント投稿済み・`議論中` + `assignee=ユーザー` で待機中 | 正常シナリオ（PoC あり）と同一の経過 |
| ユーザー依頼 | 特定候補への追加検証を依頼するコメントを投稿する | 分岐を決定的に誘発 |

### フロー

```mermaid
sequenceDiagram
  actor U as ユーザー
  participant GH as GitHub
  participant ORC as モニター
  participant MON as architect

  U->>GH: subsystem PR に特定候補への<br>追加検証の依頼コメント + assignee 外し
  ORC-->>GH: polling（ユーザー返信 + assignee なし を検知）
  ORC->>MON: 既存セッションへ送信
  activate MON
  MON->>GH: 該当 PoC PR に<br>確認:library-poc-runner 再付与 +<br>再検証指示コメント
  deactivate MON

  Note over GH: 再検証が走り、PoC PR に<br>再検証の完了報告が投稿される
  ORC-->>GH: polling（確認ラベル + 未解決コメント を検知）
  ORC->>MON: 既存セッションへ送信<br>（台帳の監視面から architect のセッションへ届く）
  activate MON
  MON->>GH: 完了報告コメントを Resolve
  MON->>GH: subsystem PR の結果まとめを更新 +<br>議論中 付与 + assignee=ユーザー 設定
  deactivate MON
  Note over MON: 結果議論の応答ループに合流
```

### 期待値

- 該当 PoC PR の再検証指示コメント（@library-poc-runner 宛）と完了報告コメントが Resolve 済み
- subsystem PR の結果まとめコメントが追加検証の結果で更新されている
- subsystem PR に `議論中` + `assignee=ユーザー` が設定されている

## 異常シナリオ（適合候補が見つからない）

### セットアップ

| セットアップ | 説明 | 補足 |
| --- | --- | --- |
| Mock | なし（実環境で実行） | - |
| ライブラリ選定論点 | 要件（ライセンス・対応環境・性能等）を満たす候補が存在しない | 例: 商用ライセンスのみ / 対応プラットフォーム外 |

### フロー

```mermaid
sequenceDiagram
  actor U as ユーザー
  participant GH as GitHub
  participant MON as architect

  MON->>MON: 候補列挙 + 調査の結果<br>全候補が要件不適合
  MON->>GH: subsystem PR に不適合の理由 +<br>代替方針の相談コメント<br>（自作 / 要件緩和 / 設計変更）+<br>議論中 付与 + assignee=ユーザー 設定
  Note over U: ユーザーの方針判断で<br>設計の応答ループに戻る
```

### 期待値

- 相談コメント（不適合の理由 + 代替方針）が投稿され、`議論中` + `assignee=ユーザー` が設定されている

## 異常シナリオ（適合候補が見つからない・epic へ方針転換）

### セットアップ

| セットアップ | 説明 | 補足 |
| --- | --- | --- |
| Mock | なし（実環境で実行） | - |
| 相談まで完了 | 全候補が要件不適合の相談コメントに対し、ユーザーが epic レベルの方針転換を指示 | 分岐を決定的に誘発 |

### フロー

```mermaid
sequenceDiagram
  actor U as ユーザー
  participant GH as GitHub
  participant ORC as モニター
  participant MON as architect

  Note over MON: 相談コメントの投稿までは<br>異常シナリオ<br>（適合候補が見つからない）と同一
  U->>GH: subsystem PR に epic レベルで<br>方針転換する指示コメント + assignee 外し
  ORC-->>GH: polling（ユーザー返信 + assignee なし を検知）
  ORC->>MON: 既存セッションへ送信
  activate MON
  MON->>GH: epic Issue に経緯 +<br>方針転換の相談コメント<br>（@epic-conductor 宛）+<br>確認:epic-conductor 付与
  MON->>GH: subsystem PR に経緯の記録コメント +<br>assignee=ユーザー 再設定
  deactivate MON
  Note over MON: 以降の判断は epic-conductor 側で行われる<br>（subsystem の設計は保留のまま待機）
```

### 期待値

- epic Issue に `確認:epic-conductor` + 経緯コメント（@epic-conductor 宛・未解決）が付与・投稿されている
- subsystem PR は `確認:architect` + `議論中` のまま（設計 Wiki への新規 commit が存在しない）
